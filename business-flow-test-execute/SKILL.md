# 業務流程測試「實際執行」產生器

這個 skill 承接 `business-flow-test-plan` 的輸出（一份業務流程測試表），對「任何軟體專案」實際跑一遍測試，把每一條測試流程都用**真實資料、真實呼叫**執行過，而不是只列出應該測什麼。

適用情境：使用者已經有一份（或想現場產生一份）業務流程測試表，並且要求「幫我實際跑」「用真的資料驗證」「每條要有好幾組不同輸入」「把輸入輸出都列出來」。

## 與 business-flow-test-plan 的分工

| Skill | 產出 |
|---|---|
| `business-flow-test-plan` | 一份「應該測什麼」的清單（測試流程、期望呈現），欄位「結果呈現」「是否通過」預設空白/「待測」 |
| `business-flow-test-execute`（本 skill） | 針對上面清單，每條**至少 5 組不同輸入資料**真正執行（API 呼叫或等效的自動化手段），把「結果呈現」「是否通過」填成真實數據，並額外整理執行過程中意外發現的系統性問題 |

如果使用者手上還沒有測試表，先呼叫 `business-flow-test-plan` 產生一份，再用本 skill 執行。

## 同等重要原則：輸入資料與結果呈現都要寫成人看得懂的語言

跟 `business-flow-test-plan` 的「人話化」原則一樣，這個 skill 產出的「測試流程」、「輸入資料 / 操作」、「驗證位置」、「期望呈現」與「結果呈現」欄位，是給業務、客服、PM、測試人員看的驗收紀錄，**不是給工程師看的 API log**。這條原則跟安全規則一樣重要，不能因為「反正是真的執行結果」就放任寫成技術語言。

**錯誤示範（工程師語言，禁止出現在這五欄）：**
- 輸入資料：`POST /auth/login {phone:0900000000, password:admin1234}`
- 結果呈現：`200 {token, user:{role:ADMIN, shopId:61803ba1-...}}`

**正確示範（人話）：**
- 輸入資料：以「涼風冷氣清洗行」商家管理者身份，手機門號 0900000000、密碼 admin1234 登入
- 結果呈現：登入成功，系統核發有效的登入憑證，確認可以看到商家後台（主控台）

具體規則：

- 「測試流程」、「輸入資料 / 操作」、「驗證位置」、「期望呈現」與「結果呈現」五欄絕對不能出現：HTTP 方法（GET/POST/PATCH/DELETE）、API 路徑（/xxx/yyy）、原始 JSON 大括號語法、原始 status code 數字（200/403/404，改用「成功」「被拒絕」「找不到」等人話）、程式欄位名（shopId、customerPhone、isBlacklisted 等，改用畫面上看得到的名詞，例如「商家」「客戶電話」「黑名單」）、UUID 等技術識別碼。
- 具體事實（金額、時間、筆數、狀態名稱、錯誤訊息文字本身）要完整保留，不能因為人話化而漏掉關鍵資訊——錯誤訊息若是畫面會顯示的文字（如「帳號或密碼錯誤」），原文照抄即可，這本來就是人看得懂的。
- 如果需要保留技術細節方便工程師事後追查（例如對應的路由檔案、實際的 HTTP status code），寫進「備註」欄，不要混進「輸入資料 / 操作」或「結果呈現」。
- **執行測試的 subagent 在記錄每一組 variant 時，就應該直接用人話寫 `input` 與 `actualOutput`**，而不是先寫技術版本、事後再翻譯——這樣寫出來的紀錄才會像真人測試員在描述「我做了什麼」「我看到了什麼」，派工 prompt 裡務必把這條寫清楚並附上述範例。
- 合併腳本（`merge_test_results.mjs`）與產生 Excel 的腳本（`build_results_workbook.mjs`）都只是把資料排版，**不會、也不該**自動把技術字串轉成人話；人話化必須在 subagent 產生資料的當下就做好，或事後由協調者逐條檢查重寫。

## 最高原則：安全永遠優先於速度

這個 skill 會對一個「活的系統」發送大量真實請求、建立/修改測試資料。專案的資料庫**很可能同時也是別人在用的正式環境**（測試前必須先確認，不要假設）。以下規則來自實際踩過的教訓，違反任何一條都可能造成不可逆的正式環境資料異動：

1. **絕對不要對不確定是不是正式環境的資料庫執行測試。** 開始前務必先問清楚、或自行判斷目前連線的 DB 是不是專案唯一（可能就是正式）的資料庫。如果不確定，一律先詢問使用者「這是測試專用資料庫還是正式/共用資料庫？」，並提供「幫你建一個本機測試專用資料庫」的選項。
2. **一律在本機建立獨立、可拋棄的測試資料庫**，不要直接對著使用者給的正式連線字串（不論是本機、雲端、還是 docker）狂發測試流量。本機通常已裝好 Postgres/MySQL 等（用 `brew services list`、`which psql` 等確認），沒有的話用 Docker 或詢問使用者。
3. **只固定一個伺服器 port，並由協調者（你自己）親自啟動與監控**，不要讓執行測試的 subagent 有機會自己啟動伺服器程序。務必在派工前把這條寫進每個 subagent 的 prompt 最上方，當成不可協商的硬性規則：
   - 目標伺服器固定是某個 port，全程不會改變
   - 連不上就停止回報，絕對不能自己 `npm run dev` / `ts-node` / `nohup` 另開一個
   - 不要嘗試其他 port、不要嘗試任何 zeabur.app / 其他雲端網址
   - 不要修改 .env 或任何設定檔
   - 不要直接寫資料庫，只能透過呼叫 API
4. **一旦伺服器 port 中途需要更換，不要用「訊息通知」的方式叫已經在跑的 subagent 改連線目標。** 這種「執行到一半、收到訊息說要改連去别的目的地」的樣式，會被安全防護機制正確判定為疑似注入攻擊而擋下（這是好事，不要想辦法繞過）。正確做法：把伺服器**換回原本大家都認的那個 port**，而不是叫大家換去新 port。
5. **危險/不可逆操作**（清空資料、刪除商家、改手機號級聯這類）：
   - 只在測試環境裡刻意準備的「拋棄用」商家/帳號上執行，絕對不要對著承載其他測試組資料的主要測試商家執行
   - 執行前務必自己讀過對應程式碼，確認需要的二次確認參數（例如要求輸入店名才解鎖）
6. **每批派工後，協調者自己要驗證環境健康**：確認伺服器還活著（用一個測試環境專屬、正式環境不可能存在的帳號登入來當「判別依據」，而不是只看 HTTP 200）、確認沒有出現預期外的伺服器程序（`ps aux | grep ts-node` 之類）。發現任何一次意外，立刻停下所有還在跑的 subagent，查清楚才能繼續。
7. 如果懷疑正式環境可能已經被誤觸（例如某個 subagent 一度連錯資料庫），**在繼續之前，先徵求使用者同意再去正式環境做唯讀查核**，查核只能是 SELECT，絕對不能寫入。查完要不是「有汙染」要立刻誠實告知使用者，「沒有汙染」也要讓使用者知道你查過了、結果如何。
8. 派工規模：**批次進行、每批 1–3 個 subagent**，不要一次全部（例如 9~10 個）平行射出去——一旦派工內容需要中途修正（換 port、補資料庫缺漏），越多平行 agent 代表越多份「舊指令」在跑，越難收拾。

## 執行流程

### 1. 取得測試計畫

讀取 `business-flow-test-plan` 產生的 JSON（如果沒有現成 JSON，從既有的 xlsx 用程式或直接請使用者提供原始 JSON），取得每條測試的 `id / section / category / flow / input / verify / expected / priority / moneyFocus / notes`。

### 2. 建立乾淨的測試環境

- 確認目標系統技術棧（看 `package.json`、`docker-compose.yml`、`.env.example` 等），找出啟動方式與資料庫連線設定方式。
- 在本機建立一個獨立的測試資料庫（例如 `<project>_test`），跑過 migration，跑過官方 seed 腳本。
- **視測試計畫需求，額外準備補充測資**：多角色帳號（不同權限層級各一組，含「全權限」與「零權限」對照組）、多租戶/多商家（至少一個「乾淨空的」用於危險操作正向案例、一個「有資料」用於危險操作負向案例）、邊界資料（例如已達到某個數量上限的紀錄、即將到期的紀錄、黑名單/停用狀態的紀錄）、有效/過期/已使用的邀請連結或 token 等。這些用專案本身的 ORM/DB client 寫一支一次性 fixture 腳本建立，不要用假的 SQL 硬插導致跟應用層邏輯（如密碼雜湊、關聯表）對不上。
- 用固定 port 啟動一份伺服器（背景執行、與目前終端機 session 解耦，例如 `nohup ... & disown`，避免因為某些環境下背景任務的生命週期跟著對話輪次被回收而意外中止）。
- 用一個「只有在這個測試庫才存在」的帳號登入，驗證真的打到你要的環境，而不是打到專案裡其他同時在跑的服務、或不小心 fallback 到 .env 裡的正式連線字串。

### 3. 分組派工

把測試項目依功能相關性分組（每組 8–15 條），每組一個 subagent，內容包含：

- 第 4 節「安全永遠優先於速度」的完整規則（複製貼上到每個 subagent 的 prompt 最上方）
- 固定的伺服器位址、相關帳號與密碼、資料庫存取方式（唯讀查詢用）
- 讀取測試計畫 JSON 取得這組要測的 id 清單
- 相關原始碼位置（路由/handler 檔案路徑），要求 subagent **先讀程式碼確認真實的請求格式**，不要用猜的
- 明確要求：每條至少設計 **5 組有意義、彼此不同的輸入資料**（正常值、邊界值、不同角色、不同金額、錯誤輸入等），真的呼叫執行，記錄：
  - `input`：**用人話描述**這組測試做了什麼操作、輸入了什麼資料（例如「以黑名單客戶的身份嘗試建立新訂單」），不要寫 HTTP 方法/路徑/JSON
  - `actualOutput`：**用人話描述**實際觀察到的結果（例如「訂單成功建立，系統沒有阻擋」），不要寫 status code/原始 JSON；錯誤訊息若本來就是畫面文字則照抄
  - `pass`：true/false（防呆情境被正確擋下也算 true）
  - `note`：簡短補充，尤其 `pass:false` 要說明差異；技術細節（路由檔案、實際狀態碼等）也寫在這裡，不要混進 input/actualOutput
- 純前端/純 UI 互動（顏色、拖曳、版面、Tab 分頁）無法用 API 驗證的項目，要求如實標記 `automated:false` 並附 `manualReason`，不要瞎猜或假造
- 輸出成獨立的 JSON 檔（每組一個檔名，例如 `results-g1.json`），格式：

```json
[
  {"id": "BF-001", "automated": true, "variants": [
    {"input": "...", "actualOutput": "...", "pass": true, "note": ""},
    ...（至少5組）
  ]},
  {"id": "BF-XXX", "automated": false, "manualReason": "..."}
]
```

小批次（1–3個）派工，每批完成後檢查環境健康再派下一批（見上方規則8）。

### 4. 處理過程中的環境問題

如果某組回報「大量 403 / 存取被拒」這類環境性錯誤（例如缺少某張關聯表的紀錄、缺少某個初始化步驟），**由協調者親自** 用專案的 ORM/DB client 診斷並修好（唯讀查詢問題所在、寫入必要的修復資料），不要要求 subagent 自己繞過或直接寫資料庫。修好後用「恢復 (resume) 同一個 subagent」的方式讓它拿著已有的上下文重跑一次，而不是重新派一個新的、從零開始的 agent。

### 5. 彙整結果

寫一支合併腳本（可直接沿用 `scripts/merge_test_results.mjs`，依專案調整路徑）：

1. 讀取原始測試計畫 JSON 與所有 `results-*.json`
2. 檢查每個測試 id 都有對應結果、沒有遺漏、沒有重複
3. 對每條測試：
   - 若 `automated:false`：輸出一列，「結果呈現」= manualReason，「是否通過」=「需確認」
   - 若 `automated:true`：把 N 組 variants 各自展開成一列（編號 `BF-001-1`…`BF-001-5`），「輸入資料/操作」= 真實送出的資料，「結果呈現」= 真實收到的回應，「是否通過」= 通過/失敗
4. 統計總數、通過/失敗/需確認數，供摘要頁使用

### 6. 整理「重大發現」

測試過程中經常會意外挖到超出原測項範圍的系統性問題（資安漏洞、時區/邊界值錯誤、缺乏輸入驗證等）——這些比單純的通過/失敗更重要，不要讓它們被淹沒在幾百列的明細裡。為每一個這類發現建立一筆：`severity`（🔴嚴重／🟠中等／🟡低）、`title`、`relatedIds`、`description`、`repro`（重現步驟）、`impact`、`recommendation`。

**發現 🔴 嚴重等級的問題（尤其是資安漏洞、會影響正式環境的問題）時，不要等整份報告做完才提——立刻停下來，用清楚的文字向使用者說明問題、如何重現、影響範圍，並詢問是否要先處理再繼續跑測試。**

### 7. 產生最終 Excel

沿用 `business-flow-test-plan` 的欄位與樣式規範（見該 skill 的「Excel 樣式標準」），額外新增一張「重大發現與建議」工作表。用 `scripts/build_results_workbook.mjs` 產生（同樣因為 `@oai/artifact-tool` 在 Claude Code CLI 環境中不存在，改用專案既有或另外安裝的 `exceljs` 套件）。

輸出到 `outputs/business-flow-test-plan/` 或使用者指定的位置，檔名建議加上「_實測結果」字樣以區別純規劃版本。

### 8. 驗證與收尾

- 檢查工作簿存在、四張工作表都在、資料列數與統計數字兜得起來、公式錯誤掃描沒有命中
- **抽查至少 10 列的「輸入資料 / 操作」與「結果呈現」，確認全是人話**（沒有 HTTP 方法/路徑/JSON/status code/程式欄位名）；若 subagent 交回的是技術版本，回頭修正後再產生 Excel，不要帶著技術語言直接交付
- 確認測試環境（伺服器程序、port）狀態，跟使用者說清楚目前是否還在背景執行、要不要幫忙關掉
- 回覆使用者：整體通過/失敗數字、最重要的重大發現（尤其 🔴 嚴重項目要放在最前面）、Excel 連結

## 完成標準

- 每條測試都有真實執行紀錄或明確標記「為何無法自動化」，沒有任何一條是憑空捏造的結果
- 每條自動化測試都至少有 5 組彼此有意義不同的輸入資料
- 「輸入資料 / 操作」與「結果呈現」兩欄全程是人看得懂的語言，抽查任一列都不該出現 HTTP 方法、API 路徑、原始 JSON、程式欄位名或裸露的 status code
- 執行過程中發現的系統性問題（不只是原測項本身的通過/失敗）都被獨立整理成「重大發現」，並依嚴重程度排序
- 全程沒有動到正式/共用資料庫；如果懷疑動到了，已經誠實跟使用者說明並在授權後完成查核
- 沒有殘留來路不明的伺服器程序
