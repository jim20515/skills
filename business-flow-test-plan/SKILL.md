# 業務流程測試表產生器

這個 skill 用來為「任何軟體專案」建立人可以直接執行的業務流程測試 Excel。它不能依賴某個既有專案的檔案、路由、角色、資料表或測試案例；必須從目前目標專案的文件、畫面、API、程式碼、既有測試、截圖、試算表或使用者描述中重新整理。

## 最重要原則

- 不要硬套本 skill 裡的範例字眼。
- 不要因為參考清單出現某個面向，就假設目標專案一定有那個功能。
- 只有當目標專案真的存在該功能時，才把它列成測試項目。
- `測試流程` 和 `期望呈現` 要寫給業務、客服、PM、測試人員看得懂，不要寫成工程師才懂的欄位名或 API 名。
- 技術資訊可以放在 `輸入資料 / 操作`、`驗證位置` 或 `備註`，但不能取代人話描述。

## 輸出格式

產出一份 `.xlsx`，至少包含三張工作表：

1. `摘要`
2. `完整測試清單`
3. `使用說明`

`完整測試清單` 欄位固定如下：

| 欄位 | 說明 |
|---|---|
| 編號 | 穩定測試編號，例如 `BF-001` |
| 大項 | 人看得懂的大分類，例如依目標專案整理出的主要流程 |
| 分類 | 更細的功能、頁面或模組 |
| 金額重點 | 牽涉金額、帳務、付款、報表等高風險數字時填 `是` |
| 優先級 | `P0 金額`、`P0`、`P1`、`P2` |
| 測試流程 | 人話流程，不要是 API 或變數名稱 |
| 輸入資料 / 操作 | 實際要輸入的資料、點擊動作、API 操作或前置條件 |
| 驗證位置 | 要去哪裡看結果，例如頁面、通知、報表、匯出檔或 API 回應 |
| 期望呈現 | 驗收者應該看到的結果 |
| 結果呈現 | 實測後填入實際看到的內容；未測前填 `待測` |
| 是否通過 | `待測`、`通過`、`失敗`、`需確認` |
| 問題位置 | 失敗時填頁面、功能或欄位 |
| 建議覆蓋方式 | `人工`、`API`、`E2E`、`單元測試` 或組合 |
| 備註 | 假設、風險、清理方式、資料限制 |

## 使用流程

1. 收集目標專案脈絡
   - 讀目前專案的 README、需求文件、路由、頁面、元件名稱、測試檔、截圖、試算表、API 文件、資料模型和使用者補充說明。
   - 優先使用目標專案真正出現在畫面上的文字、頁面名稱、按鈕名稱、狀態名稱。
   - 如果資訊不足，可以先產出第一版，但不確定的地方要標 `需確認`。

2. 找出目標專案自己的業務生命週期
   - 先辨識核心業務物件，例如目標專案裡最重要、會被建立/修改/追蹤的資料。
   - 再依該專案實際功能整理建立、查詢、修改、狀態變更、取消、刪除、復原、審核、匯入、匯出、通知、權限等流程。
   - 如果目標專案沒有某個功能，不要列入測試表。

3. 建立測試清單
   - 覆蓋正常流程、異常流程、權限、防呆、跨頁一致性、資料同步、報表與匯出、設定變更後的影響。
   - 金額、帳務、付款、退款、報表統計、分潤、點數、庫存價值等會影響錢或重要數字的項目，一律標 `P0 金額`。
   - 對沒有把握的業務規則，填 `需確認`，不要自行假設為通過。

4. 人話化
   - 將技術欄位翻成業務語言。
   - 例如「某個編號欄位」應寫成目標專案畫面上的實際名稱。
   - 例如「狀態代碼」應寫成畫面上的狀態文字，如「待審核、已核准、已退回」。
   - 例如「金額欄位」應寫成畫面或報表上的名稱，如「應收金額、付款金額、折扣後金額」。
   - 不要在 `測試流程`、`期望呈現` 裡留下 `id`、`createdAt`、`status`、`totalAmount`、`POST /xxx` 這類工程語言，除非該字本來就是畫面文字。

5. 產生 Excel
   - 建立符合下方 schema 的 JSON。
   - 用 `scripts/build_business_flow_workbook.mjs` 產出 `.xlsx`。
   - 最終檔案通常放在目前專案的 `outputs/business-flow-test-plan/`。

6. 驗證
   - 檢查工作簿存在。
   - 檢查列數正確。
   - 掃描公式錯誤。
   - 渲染 `完整測試清單` 前幾列，確認表頭、交錯列底色、狀態顏色可讀。

## 候選檢查面向

以下只是「候選面向」，不是固定清單。使用前必須先判斷目標專案是否真的有這些功能。

- 使用者進入系統、登入、註冊、登出、密碼、身分或角色
- 權限、角色、組織、租戶、資料隔離
- 核心資料的新增、查詢、修改、刪除、復原
- 狀態流轉、審核、退回、取消、完成
- 列表、詳情、搜尋、篩選、排序、分頁
- 跨頁或跨角色看到同一筆資料時的一致性
- 通知或訊息：只有專案有站內通知、Email、簡訊、Webhook、即時同步等功能時才列
- 金額或重要數字：只有專案有付款、帳務、折扣、稅、退款、庫存價值、點數、報表等功能時才列
- 報表、統計、匯入、匯出：只有專案有這類功能時才列
- 設定、基礎資料、啟用/停用：只有專案有設定或主檔維護時才列
- 排程、時間區間、跨日、時區：只有專案有時間、預約、排班、期限或排程時才列
- 外部整合：只有專案有第三方系統或外部服務串接時才列
- 錯誤、防呆、必填、重複、衝突、邊界值
- 歷史資料是否會被新設定或新規則錯誤改寫

## JSON 輸入格式

建立一個 JSON 檔，供腳本產生 Excel：

```json
{
  "title": "完整業務流程測試表",
  "generatedDate": "2026-07-04",
  "rows": [
    {
      "id": "BF-001",
      "section": "核心流程",
      "category": "主要功能",
      "moneyFocus": false,
      "priority": "P0",
      "flow": "使用者完成目標專案的一個核心操作後，相關頁面都要顯示同一筆資料",
      "input": "依目標專案的表單輸入必要資料並送出",
      "verify": "目標專案中會顯示該資料的頁面或報表",
      "expected": "畫面顯示的名稱、狀態、時間、負責人與其他關鍵欄位都和送出內容一致",
      "actual": "待測",
      "status": "待測",
      "issueLocation": "",
      "coverage": "E2E",
      "notes": ""
    }
  ]
}
```

必填欄位：

- `id`
- `section`
- `category`
- `flow`
- `input`
- `verify`
- `expected`

選填欄位預設：

- `moneyFocus`: `false`
- `priority`: `moneyFocus=true` 時預設 `P0 金額`，否則 `P1`
- `actual`: `待測`
- `status`: `待測`
- `issueLocation`: 空白
- `coverage`: 空白
- `notes`: 空白

## 執行腳本

使用 `load_workspace_dependencies` 找到 bundled Node 和 Node modules，然後執行：

```bash
mkdir -p /tmp/business-flow-skill-run
ln -sfn "$NODE_MODULES_FROM_LOAD_WORKSPACE_DEPENDENCIES" /tmp/business-flow-skill-run/node_modules
cd /tmp/business-flow-skill-run
"$NODE_FROM_LOAD_WORKSPACE_DEPENDENCIES" /path/to/business-flow-test-plan/scripts/build_business_flow_workbook.mjs \
  --input /path/to/input.json \
  --output /path/to/outputs/business-flow-test-plan/完整業務流程測試表.xlsx
```

## Excel 樣式標準

- 表頭：淺藍底、深藍粗體字。
- 資料列：白底 / 淺藍底交錯。
- `金額重點` 與 `P0 金額`：黃底、棕色粗體字。
- `通過`：綠底、深綠粗體字。
- `失敗`：紅底、白色粗體字。
- `需確認`：黃底、棕色粗體字。
- `待測`：藍底、深藍粗體字。
- 凍結表頭列。
- 長文字換行。
- 欄寬要讓人能閱讀，不要只適合機器解析。

## 完成標準

回覆使用者前確認：

- Excel 檔案已產出。
- `摘要`、`完整測試清單`、`使用說明` 三張工作表存在。
- 測試項目筆數合理。
- 公式錯誤掃描沒有命中。
- 預覽畫面可讀。
- 最終回覆只提供 `.xlsx` 連結，除非使用者要求看支援檔。
